우리 팀의 어드민(EV 충전소 관리 시스템, 약 10만 줄)을 측정해보면 이런 상태였습니다. ESLint가 설정 오류로 아예 실행이 안 되고, 테스트는 전체에 1개, `any`가 328번, 1000줄 넘는 파일이 3개. 가드레일이 통째로 빠진 레거시예요.

여기에 손을 대야 했습니다. 마침 AI 에이전트로 코드를 다루는 게 현실적인 선택지가 된 시점이었고요. 그런데 막상 "이 레거시 좀 정리해줘"라고 통째로 던지려니 손이 멈췄습니다. 가드레일이 없는 코드에 자동화된 대규모 편집이 가능한 AI를 풀어놓으면 고쳐지는 것보다 깨지는 게 더 빠를 수 있겠다는 직감 때문이었어요.

그래서 코드를 짜기 전에 먼저 작업할 판을 설계했습니다. 이 글은 그 판, 제가 `h5s`라고 부르는 에이전트 하네스를 왜 이렇게 설계했는지의 기록입니다. 실제로 돌려서 무엇이 좋아졌는지는 [다음 글](/posts/agent-harness-refactor-measured)에서 측정과 함께 다룹니다.

용어를 하나 짚고 시작할게요. 여기서 하네스는 에이전트 실행 루프 자체(도구 호출, 컨텍스트 관리 같은 것들. 그건 Claude Code가 제공합니다)가 아니라, 그 위에서 에이전트가 안전하고 검증 가능하게 일하도록 잡아주는 구조를 말합니다. 역할 정의, 검토 게이트, 측정, 운영 규칙의 묶음이요.

## 왜 그냥 "고쳐줘"가 아니었나

에이전트 하나에게 큰 리팩토링을 통째로 맡기는 건 매력적입니다. 빠르고 셋업이 없으니까요. 그런데 가드레일 없는 레거시에서는 네 가지가 걸렸습니다.

컨텍스트 오염. 한 대화 안에서 분해, 구현, 검토를 다 시키면 검토하는 시점의 에이전트는 이미 자기가 방금 짠 코드에 설득당해 있습니다. "이거 괜찮아?"라고 물으면 "네 괜찮아요"가 나와요. 구현한 주체가 자기 구현을 검토하면 객관성이 사라집니다.

통제 지점의 부재. 자동 파이프라인은 한 번 돌기 시작하면 사람이 끼어들 지점이 없습니다. 잘못된 방향으로 30분을 달린 뒤에야 결과를 보게 돼요.

폭주. 테스트를 고치려다 실패하고, 실패를 고치려다 또 실패하는 루프에 빠지면 종료 조건이 없습니다.

토큰. 대화가 길어질수록 모든 단계가 같은 누적 컨텍스트를 통째로 들고 다닙니다. 게다가 이 레거시에는 1000줄 넘는 파일이 셋이라, 그걸 매번 통째로 읽으면 그 자체가 비용이에요.

정리하면 속도를 주는 "한 방에 맡기기"가, 정작 레거시에서 제일 필요한 객관성, 통제, 종료조건, 비용 관리를 다 포기하게 만듭니다. 그래서 그 넷을 되찾는 구조를 먼저 만들기로 했어요.

## 모드 전환에서 역할 분리로

처음 우리 `CLAUDE.md`는 한 에이전트가 Orchestrator / Decomposer / Worker "모드"를 바꿔가며 일하는 것처럼 쓰여 있었습니다. 그런데 모드 전환은 결국 같은 컨텍스트 안에서 모자만 바꿔 쓰는 것입니다. 위의 컨텍스트 오염 문제가 그대로 남아요.

그래서 역할을 독립된 에이전트로 분리했습니다. 핵심은 서브/멀티 에이전트가 부모의 대화 맥락을 상속받지 않고 빈 컨텍스트로 시작한다는 점이에요. 검토자는 구현자의 사고 과정을 모른 채 오직 명세와 결과물만 보고 판정합니다. 객관성과 토큰 효율을 동시에 얻는 지점이 여기입니다.

여기서 한 가지 구분이 중요했습니다. 역할(Orchestrator, Planner, Worker 등)은 `.claude/agents/*.md`의 서브에이전트로 둡니다. 독립 컨텍스트로 실행되고 위임으로 호출되는, 우리가 원한 "에이전트"는 skill이 아니라 subagent거든요. 반면 "테스트 작성 기준" 같은 참조 지식은 실행 주체가 아니라 로드해서 읽는 지침이라 `test-strategy` skill로 뒀습니다.

이 둘을 헷갈리면 "에이전트를 skill로 만들었는데 왜 독립 실행이 안 되지?" 같은 데서 막힙니다. 실행 단위는 subagent, 참조 지식은 skill. 이 경계가 설계의 첫 단추였습니다.

## 역할 지도

최종 구조는 이렇습니다.

```
Orchestrator                         바깥 루프 컨트롤러 · 최종 go/no-go
 │   ↕ 매 단계 Verifier(사실) → 필요시 Critic(판단) 호출 → 판정 받고 집행
 ├─ [멀티에이전트] Planner ⟷ Worker
 │                          └─ [서브에이전트] Implementer → Test Author → Debugger
 ├─ Verifier   검토형 공유 평가자 · 사실 검증(테스트/빌드/lint) · sonnet
 ├─ Critic     판단형 공유 평가자 · 디자인/구조/네이밍 품질 판단 · opus
 └─ Reflector  종료 후 1회 · 교훈 → 하네스 개선 "제안"
```

| 역할             | 책임                                                      |
| ---------------- | --------------------------------------------------------- |
| **Orchestrator** | 흐름을 돌리고 **최종 진행 결정**만. 코드는 직접 안 씀     |
| **Planner**      | 큰 작업을 독립 태스크 카드로 분해 (Worker와 같은 레벨)    |
| **Worker**       | 태스크 하나의 완성을 책임지는 코디네이터                  |
| └ Implementer    | 요구사항 구현만                                           |
| └ Test Author    | 테스트 작성만 (`test-strategy` 기준)                      |
| └ Debugger       | 테스트 실패의 **원인 진단**과 수정만                      |
| **Verifier**     | 사실 검증(테스트·빌드·lint 통과, 완료기준). sonnet        |
| **Critic**       | 판단(디자인·구조·네이밍 품질). Verifier PASS 후에만. opus |
| **Reflector**    | 종료 후 교훈을 모아 하네스 개선을 **제안**                |

Orchestrator가 코드를 한 줄도 안 쓰는 건 의도적입니다. 손을 대는 순간 "내가 짠 거니까 괜찮아"라는 편향이 최종 결정에 끼니까요. 컨트롤러는 판단만 하고 구현은 다른 손이 하게 분리했습니다.

### Worker를 셋으로 쪼갠 이유

Implementer / Test Author / Debugger를 굳이 나눈 건 이 레거시의 상태와 직결됩니다. 테스트가 사실상 없는 코드(전체 1개)에 회귀 안전망을 깔아야 하고, 대형 파일 분할은 "테스트 먼저, 분할 나중" 순서가 필수예요. 구현과 테스트를 다른 역할로 두면 구현자가 자기 코드를 통과시키려고 자기충족적인 테스트를 쓰는 걸 막을 수 있습니다. 테스트는 명세 기준으로 별도 역할이 작성하니까요. 그리고 디버깅이 "왜 실패했는가" 분류부터 강제되어, 테스트를 무력화해서 통과시키는 안티패턴도 막힙니다.

자기가 짠 코드를 자기가 짠 테스트로 자기가 통과 판정하는 것. 레거시 자동화에서 가장 흔하고 가장 위험한 동작인데, 역할을 쪼개는 것만으로 이걸 구조적으로 막을 수 있었습니다.

## 에이전트끼리는 파일로 소통한다

서브에이전트가 부모 대화를 상속받지 않는다는 건 객관성에는 좋지만, 그러면 서로 어떻게 정보를 주고받느냐는 문제를 낳습니다. 답은 대화가 아니라 파일이에요.

`h5s/agent/` 폴더가 모든 에이전트의 공유 칠판입니다. 쓰기 소유권을 역할별로 못박아서 충돌을 막았습니다.

| 파일/폴더  | 누가 쓰나         | 내용                            |
| ---------- | ----------------- | ------------------------------- |
| `STATE.md` | Orchestrator      | 현재 단계·진행 현황·산출물 위치 |
| `tasks/`   | Planner           | 태스크 명세 카드                |
| `results/` | Worker 계열       | 작업 완료 보고 카드             |
| `reviews/` | Verifier / Critic | 판정 카드 (PASS/FIX/BLOCK)      |
| `history/` | Orchestrator      | 의사결정·사용자 피드백 로그     |

운영 규칙은 토큰을 아끼는 방향으로 잡았습니다. 하위 에이전트를 부를 때 내용을 붙여넣지 말고 "읽을 파일 경로"만 넘긴다. 파일은 작고 구조화되게 유지해 각자 자기 몫만 읽게 한다. 반환 메시지도 토큰이므로 `"result-002.md 작성, 테스트 5/5 통과"` 정도로 짧게 반환하고 상세는 파일에 남긴다. 그리고 한 파일은 한 역할만 쓴다(위 소유권 표).

> 여기서 정정할 게 하나 있습니다. 처음엔 이 구조를 "토큰 절약"으로 표현했는데, 멀티에이전트는 단일 에이전트보다 토큰을 대체로 더 씁니다(cold-start, 호출/반환 오버헤드). 파일 메모리가 줄이는 건 부모 대화를 통째로 복제하는 낭비이지 총량이 아니에요. 진짜 가치는 토큰이 아니라 검토 객관성, 분해와 병렬성, 사람 통제입니다. 그래서 멀티에이전트는 큰 작업에만 값어치가 있고 작은 작업엔 오히려 사치인데, 이게 바로 다음 항목으로 이어집니다.

## 사람이 끼어드는 시점: 게이트, 경중, 루프 가드

앞에서 꼽은 "통제 지점 부재"와 "폭주"를 위한 장치들입니다.

작업 경중 판단(fast / full path). 모든 작업에 풀 파이프라인을 돌리면 오타 하나 고치는 데도 토큰과 시간이 폭발합니다. 그래서 Orchestrator가 0단계에서 경중을 봅니다. 단일 파일, 국소 수정, 명확한 버그면 Planner와 멀티에이전트를 생략하고 Worker 단독 + Verifier 1회로 처리하고(fast-path), 다중 파일이나 구조 변경, 회귀 위험이 크면 전체 파이프라인을 탑니다(full-path). 애매하면 full로 가게 했어요.

사람 검토 게이트(HITL). 자동 파이프라인의 가장 큰 약점은 사람이 끼어들 지점이 없다는 것이었죠. 그래서 두 곳에서 멈춥니다. Planner가 계획을 짠 직후, 그리고 각 단계 Verifier가 PASS를 낸 직후. 멈출 때는 왜 이렇게 계획했는지 혹은 왜 통과인지의 근거를 같이 제시해 승인이나 피드백을 받습니다. `review_mode`(STEP / MILESTONE / AUTO)로 멈추는 빈도를 조절하고요.

루프 가드. 검토와 수정을 오가는 루프에서 무한 FIX 루프는 위험합니다. 같은 태스크의 FIX와 디버깅을 최대 3회로 제한하고, 넘으면 자동 재시도를 멈추고 사람에게 에스컬레이션합니다.

셋이 공통으로 말하는 건 하나입니다. 자동화의 속도를 살리되, 방향을 트는 핸들과 멈추는 브레이크는 사람 손에 남긴다.

## 사실 검증과 품질 판단을 나누다: Verifier와 Critic

평가를 둘로 쪼갠 게 이 하네스에서 개인적으로 제일 마음에 드는 결정입니다.

Verifier는 사실로 확인되는 것만 봅니다. 테스트, 빌드, lint가 통과하는가. 완료 기준을 충족했는가. 임시 주석이 남아 있지 않은가. 기계적인 일이라 sonnet으로 싸고 빠르게 돌립니다. Critic은 판단이 필요한 것을 봅니다. 디자인 시스템을 잘 지켰는가, 추상화와 구조가 적절한가, 네이밍이 일관적인가. 이쪽은 opus입니다.

굳이 나눈 이유가 있습니다. 서브에이전트의 model은 frontmatter에 고정이라 한 에이전트가 상황에 따라 상위 모델로 갈아탈 수 없어요. "판단에는 더 좋은 모델을 쓰고 싶다"를 실현하려면 역할을 나누는 게 유일한 방법이었습니다. 순서도 정했어요. 싸고 빠른 Verifier(사실 게이트)를 먼저 돌리고, PASS이면서 판단거리가 있을 때만 Critic을 부릅니다. rename이나 오타 같은 fast-path는 Critic을 생략해 비용을 아낍니다.

여기서도 판정과 집행을 분리한 게 핵심입니다. Verifier와 Critic은 PASS/FIX/BLOCK 판정만 내리고, 그 판정대로 다음 단계를 집행하는 건 컨트롤러(바깥은 Orchestrator, 안쪽은 Worker)예요. 둘 다 "내가 루프 주인"이라고 생각하면 충돌하니까 평가는 한 곳, 통제는 다른 곳으로 못박았습니다.

## graphify로 토큰 절약하기

토큰을 가장 많이 먹는 건 멀티에이전트 구조가 아니라 1000줄 넘는 파일을 통째로 읽는 행위입니다(이 레포는 1000줄+ 3개, 500줄+ 37개). 에이전트끼리 파일로 소통하도록 해서 토큰을 아낀다고 했지만 멀티에이전트가 애초에 토큰을 더 쓴다고 앞서 언급했습니다. 진짜 절약은 다른 데서 와야 했어요.

그래서 코드를 질의 가능한 지식 그래프로 인덱싱하는 graphify를 도입했습니다. 파일을 통째로 읽기 전에 그래프에 먼저 묻고, 위치와 관계를 좁힌 뒤 필요한 구간만 읽게 하는 규칙이에요.

```bash
graphify extract .                       # 코드 인덱싱 (AST 기반, API 키 불필요)
graphify query "무엇이 auth와 db를 잇나?"  # 자연어 질의
graphify explain "RateLimiter"            # 심볼 위치·역할
graphify path "UserService" "DatabasePool" # 의존/영향 경로
```

한 가지 신경 쓴 건 graphify가 `CLAUDE.md`와 git 훅을 자동으로 수정한다는 점이었습니다. 우리 하네스는 "하네스가 자기 자신을 무단으로 고치지 않는다(propose-only)"를 원칙으로 잡았기 때문에, 자동 수정 대신 이 규칙을 직접 손으로 써넣었어요. 그리고 이건 아직 검증 전 가설이라, 첫 실전에서 실제로 Read 횟수가 줄었는지 측정한 뒤 유지하거나 조정한다는 조항을 달아뒀습니다.

## 아, 맞다! - 왜 이름이 h5s인가

별건 아닌데, 하네스 산출물이 여기저기 흩어지지 않게 `.claude/`(역할·skill 정의) 외의 모든 하네스 파일을 한 폴더에 모았고 그 폴더 이름이 `h5s`입니다. **h**arness에서 가운데 5글자(`arnes`)를 숫자로 접고 **s**를 남긴, k8s(kubernetes) 작명의 오마주에요. `h5s/agent/`는 런타임 메모리라 gitignore에 넣고, `h5s/decisions/`(여러 작업에 재사용할 ADR)와 설계 문서는 git으로 추적합니다.

이 이원화가 의외로 중요했습니다. 런타임 로그는 버려도 되지만, 여러 작업에 걸쳐 존중돼야 할 결정(예: "BE 엔드포인트에 종속된 네이밍은 FE 단독으로 바꾸지 않는다")은 영속 지식으로 승격해서 다음 작업의 Planner가 읽고 따르게 했어요. 여러 사람이 하나의 에이전트를 쓰는 것처럼, 에이전트끼리 작업 컨텍스트를 이어받게 하려는 의도였습니다.

## 정리, 그리고 다음 글

가드레일 없는 10만 줄 레거시에 자동화를 풀어놓기 전에 코드보다 작업할 판(하네스)을 먼저 설계했습니다. "한 방에 맡기기"가 포기하게 만드는 객관성, 통제, 종료조건, 비용을 되찾기 위해서요. 핵심은 역할을 독립 컨텍스트의 멀티/서브에이전트로 분리한 것입니다. Orchestrator는 판단만, 구현/테스트/디버깅은 서로 다른 에이전트가, 평가는 사실(Verifier·sonnet)과 판단(Critic·opus)으로 또 한 번 분리. 에이전트끼리는 대화가 아니라 파일로 소통하고, 사람은 게이트와 경중 판단, 루프 가드로 핸들과 브레이크를 쥡니다. 진짜 토큰 절약은 멀티에이전트가 아니라 큰 파일 통독을 피하는 graphify고요.

그리고 가장 중요한 것은 검증 전엔 전부 가설입니다. 이 설계가 좋아 보이는 것과 실제로 효과가 있는 것은 다른 문제예요. 그래서 [다음 글](/posts/agent-harness-refactor-measured)에서는 이 하네스를 실제로 돌려 ESLint, 테스트, `any`, 대형 파일이 측정상 얼마나 좋아졌는지, 그리고 무엇이 설계대로 동작했고 무엇은 한 번도 실행되지 않았는지까지 숫자로 따라갑니다.
