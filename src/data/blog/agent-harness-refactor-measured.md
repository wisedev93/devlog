[이전 글](/posts/agent-harness-for-legacy)에서 코드를 짜기 전에 작업할 판 — `h5s` 에이전트 하네스 — 을 먼저 설계했다고 했습니다. 마지막 줄에 이렇게 적었어요. _"검증 전엔 전부 가설이다."_

이 글은 그 가설을 실제로 돌려본 기록입니다. EV 충전소 관리 어드민(약 10만 줄, 689개 파일)을 하네스로 리팩토링했고, 모든 단계를 `measure.sh`로 before/after를 재서 `REPORT.md`에 박았어요. 결과부터 말하면 **코드 품질은 측정 가능하게 좋아졌습니다.** 다만 이 글에서 더 하고 싶은 이야기는 "얼마나"가 아니라 **"무엇이 그 좋아짐을 만들었나"** 예요.

## Table of contents

## 먼저, 무엇을 쟀나

리팩토링에서 제일 빠지기 쉬운 함정은 "좋아진 것 같다"로 끝나는 거예요. 그래서 작업 전에 측정 스크립트부터 짰습니다. 거창할 것 없이, `grep`과 `find`와 `wc`로 객관적으로 셀 수 있는 것만 셉니다.

```bash title="scripts/measure.sh (발췌)"
echo "  any 사용       : $(grep -rn ': any\|as any\|<any>' --include=*.ts --include=*.tsx src | wc -l)"
echo "  console.* 호출 : $(grep -rn 'console\.' --include=*.ts --include=*.tsx src | wc -l)"
echo "  1000줄+ 파일   : $(find src \( -name '*.ts' -o -name '*.tsx' \) -exec wc -l {} + | awk '$1>=1000 && $2!="total"' | wc -l)"
# ESLint가 실제로 실행되는지까지 스크립트가 직접 확인
```

**숫자는 거짓말을 못 합니다.** 작업 전에 한 번, 작업 후에 한 번 돌리면 끝이에요. "체감상 깨끗해졌다"가 아니라 "`any`가 328에서 34로 줄었다"라고 말할 수 있게 되는 것 — 이게 이 작업에서 가장 먼저 확보한 자산이었습니다.

## 청구서가 아니라 성적표

`measure.sh` 기준 before/after입니다.

| 지표               | Before       | After                  | 평가            |
| ------------------ | ------------ | ---------------------- | --------------- |
| ESLint 실행        | ❌ 파싱 실패 | ✅ 정상                | 가드레일 복구   |
| `any` 사용         | 328          | **34** (90%↓)          | ✅              |
| `console.*`        | 385          | **187** (51%↓)         | ✅ (log만 제거) |
| 테스트 파일/케이스 | 1 / 1        | **8 / 48** (전부 통과) | ✅              |
| PascalCase 파일    | 63           | **0** (camelCase 통일) | ✅              |
| `Tabel` 오타 파일  | 2            | **0**                  | ✅              |
| 공백시작 파일/폴더 | 2            | **0**                  | ✅              |
| 1000줄+ 파일       | 3            | **1**                  | 🟨 부분         |

몇 개는 따로 짚을 만해요.

**ESLint 복구가 키스톤이었습니다.** Before의 lint는 단순히 위반이 많은 게 아니라, `eslint.config.js`에 존재하지 않는 옵션(`consistentTypeDefinition`)이 들어 있어 **설정 파싱 자체가 실패**했어요. 가드레일이 켜지지조차 않던 상태. 이걸 고쳐 lint를 다시 켠 게 이후 모든 작업의 출발점이 됐습니다.

**대형 파일 분할은 솔직히 부분 성공이에요.** `routers/index.tsx`는 1,257줄 → 49줄(15개 도메인별 라우트 파일로), `repairUpdate.tsx`는 1,060줄 → 386줄(섹션 컴포넌트 3개 + 커스텀 훅 1개 추출)로 떼어냈지만, `partnerManagementType.ts`(1,381줄)와 500줄+ 37개는 손대지 않았습니다. 다 했다고 쓰는 것보다 **어디까지 했는지를 정확히 쓰는 게** 측정 규율의 핵심이라고 생각했어요.

## "ESLint를 켜면 나머지가 쏟아진다"는 가설

작업을 T1~T5(설정 복구 → 네이밍 → 죽은코드 → 테스트 → 분할)의 5단계로 끝낼 줄 알았습니다. 핵심 가설은 하나였어요 — **ESLint가 모든 문제의 뿌리이고, 그걸 켜면 나머지가 cascade로 드러난다.**

이 가설은 **강하게 맞았습니다.** lint를 켜자 그동안 안 보이던 에러가 수천 건 쏟아졌고, 계획에 없던 후속 작업이 줄줄이 끌려 나왔어요 — lint 에러 정리, `any`/`console` 대청소(T6), 카카오맵 SDK 타입 선언, null 처리. **계획 밖 작업이 원래 본체보다 컸습니다.**

그래서 `any`를 328 → 34로 90% 줄이는 동안 이런 변환들이 따라왔어요.

| 변환                           | 건수 | 의미                                |
| ------------------------------ | ---- | ----------------------------------- |
| `\|\|` → `??` (nullish)        | 196  | falsy 전체가 아니라 null만 fallback |
| floating promise에 `void` 접두 | 142  | fire-and-forget 의도 명시           |
| `${expr}` → `${String(expr)}`  | 416  | template expression 타입 안전       |
| `console.log` 제거             | 198  | error/warn은 의도된 거라 유지       |

여기서 하나 일러둘 건, 남은 ESLint 에러 391건은 전부 `no-unsafe-*` 계열 — **API 응답에 타입을 안 준 데서 cascade된 것**이에요. 이건 엔드포인트별 Zod 스키마를 붙이는 별도 작업이라 판단해서 의도적으로 다음으로 미뤘습니다. "0으로 만들었다"가 아니라 "성격이 다른 한 덩어리로 격리했다"가 정확한 표현이에요.

## 진짜 값을 한 것 — 안전 게이트가 회귀를 잡은 순간들

여기가 이 글에서 제일 하고 싶은 이야기입니다. 위의 변환들을 보면 알겠지만, 대부분 **정규식·스크립트 기반의 대량 일괄 편집**이에요. 196건을 일일이 손으로 바꾸진 않으니까요. 그런데 **대량 일괄 편집은 위험합니다.** 그리고 그 위험을 실제로 잡아낸 게 하네스에서 설계한 안전 게이트였어요. 다음과 같은 세 장면입니다.

**장면 1 — 테스트가 의미적 회귀를 잡았다.** `||` → `??` 일괄변환을 돌렸는데, `src/utils/env.ts`는 빈 문자열(`''`) 환경변수일 때 기본값이 적용돼야 하는 곳이었어요. `??`는 `null`/`undefined`만 fallback하니까 `''`는 그대로 통과합니다 — 동작이 바뀐 거예요. 이걸 **기존 테스트가 깨지면서** 잡아냈고, 해당 파일만 `||`을 유지하고 `eslint-disable`로 되돌렸습니다.

**장면 2 — PR 리뷰 게이트가 자동화의 자해를 잡았다.** `${expr}` → `${String(expr)}` 일괄 래핑이 일부 파일에 **구문 에러를 주입**했어요. `${String(String(...))}` 같은 이중 래핑, `${String()}` 빈 호출 같은 것들. PR #298 리뷰에서 이걸 **Critical 6건**으로 잡아 복구했습니다. 같은 리뷰에서 `?.diagram.` → `?.diagram?.` 옵셔널 체이닝 누락 같은 회귀도 함께 걸렸어요.

**장면 3 — 자기 교정.** JSX 안에서 `// eslint-disable` 주석이 화면에 **텍스트로 그대로 노출되던 버그**가 52건 있었어요(`//`는 JSX에서 주석이 아니니까). 이걸 `{/* ... */}`로 바꾸고, 변환 중 잘못 건드린 15건은 스스로 되돌렸습니다.

세 장면의 공통 교훈은 분명해요 — **대량 일괄 편집은 위험하고, 그걸 잡은 건 "테스트와 사람 리뷰 게이트"였다.** 멀티에이전트 구조 그 자체가 아니라요. 자동화가 빠르게 많이 바꿀수록, 그 변경을 사후에 검증하는 게이트가 더 중요해집니다.

## 사람이 방향을 튼 순간 — HITL

게이트가 회귀만 잡은 건 아니에요. 사람 피드백이 작업 *범위*를 제때 좁힌 장면도 있었습니다.

네이밍을 통일하면서 생성/등록 화면 동사가 `Regist`(20) / `Create`(6) / `Register`(4)로 섞여 있는 걸 발견했어요. 자동화 입장에선 "하나로 통일하자"가 자연스러운 결론이죠. 그런데 게이트에서 멈추고 확인하니, **이 이름들이 BE 엔드포인트에 종속**돼 있었습니다. FE에서 단독으로 바꾸면 API가 깨져요. 그래서 동사 통일은 범위에서 빼고, PascalCase → camelCase **파일명 케이스 통일**(63개)만 진행했습니다.

이건 자동화 혼자선 내리기 어려운 판단이었어요. **"통일하는 게 옳다"는 일반 원칙과, "여기선 BE가 묶여 있으니 건드리면 안 된다"는 맥락** 사이의 선택. 사람이 끼어드는 지점을 설계에 박아둔 게 정확히 이런 데서 값을 했습니다. 그리고 이 결정은 일회성 로그가 아니라 **"BE 종속 네이밍은 FE 단독 변경 금지"** 라는 영속 규칙으로 승격해, 다음 작업이 같은 실수를 안 하게 남겼어요.

## 정직한 귀인 — 무엇이 효과를 만들었나

성과는 분명하니, 짧게 한 가지만 짚고 넘어갈게요. **"그래서 이 향상이 멀티에이전트 구조 덕인가?"** 라고 물으면, 정직한 답은 *분리해서 입증할 수 없다*입니다. A/B 테스트가 없었고, 같은 결과를 "측정과 게이트를 동반한 꼼꼼한 순차 계획"으로도 상당 부분 얻을 수 있었을 거예요.

설계했지만 한 번도 발화하지 않은 레이어도 솔직히 있었습니다 — Playwright e2e·접근성 검증은 인프라만 깔고 실행은 안 됐고, [1편](/posts/agent-harness-for-legacy)에서 기대를 걸었던 **graphify도 이번 실행에선 실제 사용 흔적이 없어** 토큰 절약 효과는 미검증으로 남았어요. 자기보고 효율 지표(`6/6 1회 통과`)도 매끄러웠지만, 실제론 T3~T5에서 재작업이 있었으니 살짝 장밋빛이었습니다.

그래서 이 실험이 보여준 건 _"멀티에이전트 하네스가 우월하다"_ 가 아니라, 더 담백한 명제예요 —

> **측정 + 안전 게이트(테스트·리뷰) + 사람 피드백을 강제하는 규율이, 큰 레거시 리팩토링에서 실질적 효과를 낸다.**

이 규율은 멀티에이전트가 아니어도 상당 부분 가져올 수 있어요. 다만 하네스가 그 규율을 **빠뜨리기 어렵게 강제**해준 건 분명한 값이었습니다. 다음 단계는 새 기능을 더 얹는 게 아니라, 안 쓰는 레이어를 정리하고 지표를 정직하게 만들어 **이미 값을 한 부분을 단단히** 하는 것 — 1편에서 설계 내내 받았던 피드백, _"좋은 하네스의 다음 덕목은 더하기가 아니라 검증과 빼기"_ 가 실행으로도 확인된 셈이에요.

## 마무리

- 가드레일 없던 10만 줄 레거시를 하네스로 리팩토링했다. ESLint **실행 불가 → 정상**, `any` **328 → 34(90%↓)**, 테스트 **1 → 48케이스**, PascalCase **63 → 0**, 1000줄+ 파일 **3 → 1**. 전부 `measure.sh`로 재현·재측정 가능.
- 핵심 가설 — _"ESLint를 켜면 나머지가 cascade로 드러난다"_ — 은 강하게 맞았고, 그 바람에 계획 밖 작업이 본체보다 커졌다.
- 그러나 이 작업의 진짜 주인공은 화려한 숫자가 아니라, **대량 일괄 편집이 심은 회귀를 테스트와 PR 리뷰 게이트가 잡아낸 순간들**이었다. 자동화가 빠를수록 사후 검증 게이트가 더 중요하다.
- 사람 피드백은 `Regist/Create/Register`처럼 **BE에 종속된 변경을 제때 제외**해 사고를 막았다.
- 정직하게 — 향상이 "멀티에이전트 구조" 고유의 기여인지는 분리·입증할 수 없다. 입증된 건 **측정 + 안전 게이트 + 사람 피드백을 강제한 규율**의 효과다. e2e·graphify 같은 일부 레이어는 이번에 발화하지 않았고, 그건 다음 작업에서 켜거나 접을 숙제로 남았다.

레거시에 AI를 들이는 일에서 제가 배운 한 줄을 남기자면 — **자동화에게 속도를 맡기되, 그 속도를 증명하고 멈출 수 있는 규율은 사람이 쥐고 있어야 한다.** 숫자로 좋아졌다고 말할 수 있다는 것, 그리고 좋아지는 과정에서 깨진 것들을 게이트가 잡아줬다는 것 — 그 둘이 이번 작업의 실제 성과였습니다.
