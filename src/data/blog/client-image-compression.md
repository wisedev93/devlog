[이전 글](/posts/direct-to-storage-upload)의 마지막 줄을 옮겨두면 — _"`4G의 100초`는 이번 작업이 옮기지 못한 시간이다. 시간을 옮기는 게 아니라 없애려면 업로드 바이트 자체를 줄여야 하고, 거기서부터는 다른 이야기다."_

이 글이 그 "다른 이야기"입니다. 같은 4G에서 98초 걸리던 업로드를 17초로 개선한 과정과, 그 사이에서 본 JPEG 압축의 내부, 그리고 우연히 발견한 블러-압축 시너지에 대한 기록입니다.

## Table of contents

## 왜 클라이언트인가 — 서버 압축이 못 푸는 것

가장 먼저 생각한 건 "서버가 받은 다음에 압축하면 되지 않나?" 였습니다. 답은 아니였습니다. 두 가지 이유로 서버 압축은 이번 문제를 풀 수 없었습니다.

첫째, **4.5MB 한도가 그대로 입니다.** [이전 글](/posts/direct-to-storage-upload)에서 서명 업로드 URL로 우회한 그 한도인데 압축은 서버가 *받은 다음*에 일어나므로, 사용자가 일단 원본 ~40MB를 서버에 보낼 수 있어야 합니다. 한도를 넘는 순간 함수는 실행조차 못 해요. 우회한 한도가 의미가 없어집니다.

둘째, **업로드 시간이 그대로 입니다.** 4G에서 4MB 사진 4장을 보내는 데 98초가 걸렸습니다. 어떻게 한도를 우회해서 서버에서 압축한들 사용자는 사진을 서버로 보내는 98초를 기다려야해요. *"서버가 압축한다"는 말은 결국 "원본 그대로 보낸다"는 뜻*이라 잘못된 접근이었어요.

그래서 **사용자의 브라우저**, 정확히는 사용자의 CPU에서 압축을 해줘야 했습니다. 사용자 한 명이 자기 폰에서 한 번 압축 비용을 내면 _모든_ 후속 단계(업로드·저장·블러)의 비용이 줄어듭니다.

선택지를 표로 두면 이렇습니다.

| 자리           | 4.5MB 한도 회피 | 업로드 시간 단축 | 비용       |
| -------------- | --------------- | ---------------- | ---------- |
| 클라이언트     | ✓               | ✓                | 사용자 CPU |
| 서버 (수신 후) | ✕ (한도 그대로) | ✕                | 서버 CPU   |
| CDN / 엣지     | ✕ (한도 그대로) | ✕                | 외부 비용  |

압축의 시점이 "근본적"으로 클라이언트인 이유 — **시간을 절약하려면 결국 절약하려는 그 시점 "이전"에 일을 끝내야 합니다.**

## 그러면 직접 업로드는 이제 필요 없나?

이때 합리적인 의문 하나가 생겼습니다. **"클라이언트에서 압축해 4장 × 0.65MB ≈ 2.6MB짜리 페이로드가 나온다면, [이전 글](/posts/direct-to-storage-upload)에서 한 작업은 불필요해진 것 아닌가? FormData로도 4.5MB 한도 안에 들어오는데?"** 하지만 잠깐 멈춰서 생각해보니 **측정값만 보면 맞는 말인데 아키텍처로 보면 직접 업로드는 그대로 필요하다고 생각했어요.**

### 1. 압축은 실패할 수 있다, 그때 *원본*이 그대로 전송된다.

`PhotoUpload`의 `catch` 블록을 보면 이렇게 되어 있어요.

```ts title="components/apply/PhotoUpload.tsx"
try {
  out = await imageCompression(f, { /* ... */ });
} catch (err) {
  // 압축 실패 → 폼은 막지 않고 원본을 그대로 전송한다 (Android Chrome 의 HEIC 등)
  compressionFailed = true;
  reportApplyError({ stage: "compress-failed", /* ... */ });
}
```

압축이 실패하면 *원본*을 그대로 다음 단계로 넘깁니다. Android Chrome에서 HEIC 디코드 실패, 저가폰의 메모리 부족, 특이 EXIF로 인한 canvas 에러, 워커 초기화 실패 등. 때문에 폴백은 필수입니다.

문제는 *그 폴백의 페이로드 크기*예요. 원본은 `PHOTO_LIMITS.maxSizeBytes = 10MB`까지 허용됩니다. 4장 모두 폴백 + 모두 10MB짜리 원본이면 **최대 40MB.** 4.5MB의 9배예요. 압축이 한 번 깨지는 순간 4.5MB 한도가 다시 살아나고, *이때 직접 업로드가 없으면 신청 자체가 통째로 실패*합니다.

이게 핵심이에요 — **압축은 *성능 최적화*이고, 직접 업로드는 *구조적 안전망*이다.** 안전망이 없으면 성능 최적화가 한 번 미끄러질 때 폼이 통째로 무너집니다. *압축이 잘 되는 best case*가 안전의 근거가 될 수는 없어요. 안전의 근거는 *압축이 실패한 worst case*를 어떻게 처리하냐예요.

### 2. 사용자와의 약속을 지키려면 *통계*가 아니라 *최악*이 기준이 되어야 한다

위의 내용을 다르게 표현하면 이렇게 됩니다.

폼은 사용자에게 "장당 10MB까지 올릴 수 있다"고 약속합니다. 그 약속을 지킬 수 있는 아키텍처여야 해요. _"보통은 압축이 잘 돌아서 0.65MB로 줄어든다"_ 같은 통계는 안전의 근거가 못 됩니다 — 압축이 깨지는 그 한 번에 약속도 같이 깨지니까요. 사용자 입장에서는 ~압축이 실패해서~ "신청이 안 됐다"가 전부일 뿐, _왜_ 안 됐는지는 관심이 없어요.

자주 부딪히는 함정이 이거예요 — **상한선을 일반적인 상황으로 정당화하는 것.** "보통 이 정도 들어오니까 괜찮다"가 _언젠가_ 깨집니다. 그리고 깨지는 시점은 보통 가장 비용이 큰 작업이예요. 4.5MB 한도 자체가 플랫폼 *고정값*이라는 사실에서 출발한 이전 글에서 그걸 우회했는데, 압축이 도입됐다고 들어내는 건 _상한선을 다시 낮추는_ 후퇴입니다. **상한선이 높을수록 사용자와의 약속을 잘 지킬 수 있습니다.**

### 3. 함수 실행 시간은 비용이다

마지막 이유는 비용 모델에서 옵니다. Vercel 서버리스 함수는 *실행 시간 × 메모리*로 청구돼요. 그리고 함수는 _요청의 본문을 다 받을 때까지 살아 있습니다._

FormData로 사진을 함수에 통과시키면 다음과 같은 일이 일어나요. 4G 사용자가 17초 동안 사진 4장을 밀어 올리는 17초 내내, 우리 함수도 17초 동안 *대기 상태*로 살아 있어요. CPU는 거의 안 쓰지만 메모리는 잡혀 있고, 시간은 카운트됩니다. 사용자 한 명에 17 GB·초가 청구되는 셈이에요.

직접 업로드는 그 17초를 *Supabase Storage 인프라*로 옮겨버립니다. 우리 함수는 1초 안에 끝나는 JSON 두 번만 처리하면 끝이에요 — `upload-urls`에서 티켓 발급, `/api/apply`에서 row insert. 둘 다 합쳐도 1초 이내. **사용자당 17 GB·초가 1 GB·초가 됩니다.** 트래픽이 작을 땐 안 보이는 차이지만, 사용자 100명만 동시에 올려도 1,700 GB·초 vs 100 GB·초 — 17배의 차이예요.

게다가 Supabase의 서명 업로드 URL은 _resumable_(TUS 프로토콜)을 지원합니다. FormData는 90%에서 연결이 끊기면 처음부터 다시 시작해야하고 이건 특히 모바일에서 의미가 큰 차이예요.

### 정리 — 다른 layer의 문제를 같이 푸는 한 쌍

| 도구            | 문제                                  | layer     |
| --------------- | ------------------------------------- | --------- |
| 직접 업로드     | 페이로드 한도 · 폴백 경로 · 함수 비용 | 구조적    |
| 클라이언트 압축 | 전송 시간                             | 성능 / UX |

두 방법이 같은 layer의 문제가 아니라 **다른 layer의 문제**입니다. 직접 업로드는 *worst case*를 받아내고, 압축은 *best case*를 빠르게 만들어요. 한쪽만 있어도 보통은 동작하지만 문제가 생기면 해결할 수가 없어요. 그래서 두 작업은 서로를 지지하는 *한 쌍*이예요.

## 어떻게 줄어드나 — 두 단계의 손실

브라우저에서 사진을 줄이는 방법은 두 가지가 함께 일어나요. 둘은 성질이 다른 손실이고, 서로 다른 방식으로 파일 크기를 줄입니다.

### 1단계 — 해상도 다운스케일

가장 단순한 손실. 4032×3024(12MP) 사진을 1920px 크기로 줄이면 픽셀 수가 약 1/6이 됩니다. *"칸 자체를 작게 만든다"*고 생각할 수 있고 작은 칸수는 작은 데이터를 의미해요.

브라우저는 이걸 `<canvas>`로 처리합니다. 라이브러리가 `Image`로 원본을 디코드하고, 더 작은 canvas에 `drawImage(..., dstW, dstH)`로 그려요. 그리는 과정에서 브라우저의 *bilinear/bicubic interpolation*이 일어나 적당히 안티앨리어싱된 다운스케일 결과가 나옵니다.

여기서의 정보 손실은 **회복 불가능**해요. 줄어든 칸 안에 원본의 픽셀 단위 정보는 더 이상 없어요. 다만 사람의 눈(폰 화면 보통 폭 700~1100px)에는 1920px 이 이미 충분해서 차이를 거의 못 느껴요.

### 2단계 — JPEG 양자화: 진짜 손실이 일어나는 시점

해상도를 줄였어도 칸 안의 값을 그대로 저장하면 여전히 큽니다. JPEG는 여기서 *진짜 압축*을 해요. 핵심 3단계는 이렇습니다.

**(1) DCT — 픽셀을 주파수로 옮기기.** JPEG는 이미지를 `8×8` 블록으로 자릅니다. 각 블록에 대해 *Discrete Cosine Transform*을 적용해, 픽셀 값들을 **주파수 성분**으로 바꿔요. 결과는 똑같이 8×8개의 숫자지만 의미가 달라집니다 — 왼쪽 위는 _낮은 주파수_(블록 전체의 평균 톤), 오른쪽 아래로 갈수록 _높은 주파수_(섬세한 디테일).

![JPEG 8×8 블록의 DCT 결과 — 좌상단이 DC(블록 평균 톤), 우하단으로 갈수록 고주파 AC](/assets/blog/client-image-compression/jpeg-dct-block.svg)

여기까지는 _손실이 없어요_. DCT는 가역 변환입니다.

**(2) 양자화 — 사람 눈이 둔감한 곳을 거칠게 깎기.** 핵심은 여기예요. 각 주파수 계수를 *양자화 테이블*의 값으로 나누고 반올림합니다. 양자화 테이블의 값은 **고주파일수록 큽니다** — 거칠게 반올림한다는 뜻이에요.

![JPEG 양자화 테이블 (q85 근처) — 저주파는 작은 값으로 정밀하게, 고주파는 큰 값으로 거칠게 양자화한다](/assets/blog/client-image-compression/jpeg-quantization-table.svg)

이 한 단계가 JPEG의 **유일한 손실**이고 압축의 정수예요. 그런데 왜 이게 _사진에서는_ 안 보일까요? **사람 눈은 고주파 변화에 둔감하기 때문입니다.** 같은 강도의 차이가 저주파에 있으면 즉시 보이고(하늘이 갑자기 어두워짐), 고주파에 있으면 잘 안 보입니다(잔디 한 가닥의 테두리). JPEG는 인간 시각의 이 *불균등한 민감도*를 인코딩 전략으로 선택했어요. 그래서 "사람이 못 느끼는 양만큼만 정보를 버리는" 압축이 됩니다.

**(3) Huffman 코딩 — 압축한 결과를 효율적으로 묶기.** 양자화된 결과는 대부분 *0*이에요(특히 고주파 영역). 0이 길게 늘어선 시퀀스를 *run-length*로 묶고, 자주 나오는 값에 짧은 코드를·드문 값에 긴 코드를 주는 Huffman 인코딩을 적용합니다. 이 단계도 _손실이 없어요_ 단지 효율적인 표기일 뿐.

`quality` 파라미터(0~1)의 진짜 의미가 이거예요. **양자화 테이블의 스케일링 계수.** 품질이 낮을수록 양자화 값을 더 크게(= 더 거칠게) 만듭니다. q85가 "안전 구간"으로 자주 추천되는 이유는 저주파 값은 여전히 작은 값으로 양자화되어 정밀하게 보존되고, 고주파만 충분히 거칠게 양자화돼 _사람이 못 느끼는_ 부분만 압축되요.

## 사람이 인식하나 — 시인성

이론은 그렇고, _실제로_ 안 보이는가? 두 가지 방식으로 봅니다.

**객관 지표 — SSIM / PSNR.** 원본과 압축본의 차이를 수치로 잴 수 있어요. 가장 흔히 쓰이는 두 지표:

- **PSNR (Peak Signal-to-Noise Ratio)** — 픽셀 값의 평균 제곱 오차 기반. 40dB 이상이면 거의 구별 불가, 30dB 이하면 명확히 보임.
- **SSIM (Structural Similarity Index)** — 사람 시각에 더 가까운 지표. 1.0이 완전 동일, 0.95+ 면 대부분의 사람이 차이를 못 봄.

JPEG q85에서 일반적인 사진은 SSIM 0.97~0.99 범위에 들어옵니다. 이게 *대부분의 사람이 차이를 못 느낀다*는 영역이고, 본문에서 `initialQuality: 0.85`를 디폴트로 잡은 근거이기도 해요.

**상황 적합성 — 어디서 보는가.** 우리 케이스는 폰으로 보는 사진입니다. 1920px 입력을 700~1100px으로 더 작은 폰 화면에서 보는 상황이에요. 이 _재축소와 같은 상황_ 자체가 추가적인 다운샘플링이라 압축 아티팩트가 한 번 더 평균화돼 사라져요. 4K 모니터 100% 확대에서 평탄한 면을 응시하지 않는 한 차이를 알아볼 일이 없습니다.

| 구간                      | 원본 평균 | 압축 후 평균 | 감소율 |
| ------------------------- | --------- | ------------ | ------ |
| 기본 사진 (`photos`)      | 3.92MB    | 0.65MB       | 83%    |
| 옵션 사진 (`optionPhoto`) | 3.92MB    | 0.65MB       | 83%    |
| 압축 자체 소요 (장당)     | —         | —            | 125ms  |

압축 자체 비용은 데스크탑 Chrome 기준 _장당 125ms_. 4장이면 500ms 정도예요. 다음 표의 단축 시간을 보면 이 500ms가 얼마나 작은 비용의 거래인지 분명해집니다.

### 측정 2 — 업로드 시간 (DevTools Network)

| 회선                | 압축 전 (median) | 압축 후 (median) | 단축    |
| ------------------- | ---------------- | ---------------- | ------- |
| no-throttle (Wi-Fi) | 1980ms           | 592ms            | **70%** |
| Fast 4G             | 98,346ms         | 17,250ms         | **82%** |

이게 핵심 숫자예요. **4G 98초 → 17초.** 압축 비용 500ms를 내고 81초를 회수한 거래 — 약 160배의 ROI입니다. Wi-Fi에서도 70% 단축. 회선에 무관하게 의미가 있어요.

[이전 글](/posts/direct-to-storage-upload)의 _병렬화 효과_ — Fast 4G에서 2%, Wi-Fi에서 21% 단축 — 와 비교하면 차이가 분명합니다. **일을 _옮기는_ 것(병렬화)은 작은 보너스, 일을 _없애는_ 것(압축)은 결정적.**

### 측정 3 — Supabase Storage 실제 저장 용량

| 버킷                                     | 장당 평균 |
| ---------------------------------------- | --------- |
| `applicant-photos` (원본 압축본)         | 634KB     |
| `applicant-photos-blurred` (블러 처리본) | 122KB     |

브라우저에서 압축한 0.65MB짜리 JPEG이 Supabase에 그대로 들어가 634KB로 저장돼요. 클라이언트가 측정한 0.65MB(=665KB)와 거의 일치 — 전송 중 손실 없이 그대로 도달했다는 뜻입니다.

그리고 _블러 처리본_ 줄. 122KB. **원본 압축본 대비 80% 추가 감소.** 처음 봤을 때 의심스러웠던 숫자고, 의심한 만큼 알아볼 가치가 있는 결과였어요.

## 블러가 파일을 또 줄인다 — 주파수 도메인의 보너스

처음 측정 결과를 보고 "이게 맞나?" 싶었어요. 634KB가 블러 한 번에 122KB로? _5배_? 잘못 측정한 줄 알고 다시 잰 숫자고, 다시 봐도 같았습니다.

설명은 위의 JPEG 압축 원리로 돌아가면 됩니다.

**가우시안 블러는 low-pass filter예요.** 정의상 *고주파 성분을 제거*합니다. 우리 코드의 `PHOTO_BLUR_SIGMA = 25`는 1920px 이미지에 적용하기엔 매우 강한 블러 — 약 75 픽셀(=3σ) 이하 디테일이 사실상 사라져요. 결과 이미지는 부드러운 색 그라데이션에 가깝습니다.

이걸 JPEG로 다시 인코딩할 때 무슨 일이 일어나나? 위의 3단계를 다시 따라가면 명확해요.

1. **DCT**: 8×8 블록의 거의 모든 픽셀이 비슷한 값이라, DCT 결과의 *AC 계수들이 거의 0*이 됩니다. DC 계수(평균값)만 의미 있는 숫자예요.
2. **양자화**: 작은 AC 값들이 큰 양자화 값으로 나누어지면서 _반올림으로 모두 0이 됩니다_. 8×8 블록 64개 칸 중 63개가 0이 되는 셈.
3. **Huffman**: 0의 *긴 런*이 RLE로 묶이고 짧은 코드로 인코딩됩니다. 0의 시퀀스가 길수록 압축률이 폭발해요.

다시 말해, **블러는 JPEG의 압축 효율을 결정짓는 "버려야 할 후보(고주파)"를 미리 다 0으로 만듭니다.** 그래서 양자화 단계가 별로 할 일이 없어지고, Huffman 단계만으로 파일이 크게 줄어요.

**어떻게 80%인가** — 이건 환경 의존적이지만 합리적 범위 안에 있어요. 단색 1920px JPEG가 5~10KB, 이 케이스의 *심하게 블러된 사진*은 그 사이 어딘가로 122KB는 그 범위의 있는 자연스러운 수치에요. 강한 블러지만 색·밝기 그라데이션은 여전히 살아있어서 완전 단색 수준까진 안 줄어들었어요.

이건 *의도한 결과가 아니라 부수효과*였습니다. 블러를 넣은 이유는 서비스 특성이었습니다. 그런데 같은 작업이 Storage 용량까지 80% 깎아주는 효과를 같이 가져왔어요. 한 작업이 여러 목적을 동시에 달성하는 드문 케이스였고 *주파수 도메인에서 보면 블러와 압축이 같은 일을 한다*는 게 그 이유였습니다.

## 트레이드오프 — 공짜는 아니다

압축이 무조건 이득은 아니에요. 당연히 비용이 있습니다.

**저가폰의 CPU 시간.** 우리 측정의 125ms는 _데스크탑 Chrome_ 기준이에요. 실제 사용자 환경은 더 느릴 수 있어요. 최근 아이폰 1~3초, 저가 안드로이드 5~10초. 4G에서 81초 아끼는 게 5~10초 압축 비용보다 큰 거래라 여전히 이득이지만, *Wi-Fi에서 1.4초 아끼려고 10초 압축*은 손해로 회선이 빠를수록 이득 폭이 줄어든다는 점을 의식해야 합니다.

이번 구현은 모든 케이스에 압축하지만, 더 정밀한 설계는 *회선·기기를 보고 압축을 건너뛰는 옵션*까지 두는 거예요. 다만 그 추가 분기의 가치가 (구현 비용 대비) 작아서 일단 단순한 설계 쪽을 택했습니다.

**HEIC on Android.** 라이브러리(`browser-image-compression`)는 마법으로 HEIC를 변환하지 않아요. 내부적으로 `Image` → `canvas.drawImage` → `canvas.toBlob('image/jpeg')` 흐름을 타는데, 이건 *브라우저가 HEIC를 디코드할 줄 알 때*에만 동작합니다. iOS Safari는 native로 HEIC 디코드를 지원 → 통과. Android Chrome은 지원 안 함 → 실패 → 원본 폴백 → `resolvePhotoKind`에서 다시 거부. 안드로이드 사용자가 HEIC를 들고 오는 일은 드물어서 이걸로 충분하지만, *모든 HEIC를 처리하는 마법은 없다*는 점은 분명해요.

**EXIF orientation.** 사진을 픽셀 단위로 재인코딩하면서 EXIF 메타데이터를 통째로 버리면, _회전 정보가 같이 사라져_ 사진이 옆으로 누워서 올라갑니다. 라이브러리가 이걸 자동 처리해줘요. orientation을 픽셀에 미리 반영(회전된 픽셀로 재배열)한 뒤 EXIF는 버리는 방식으로 직접 canvas로 짠다면 EXIF 파싱·회전이 별도 작업으로 따라옵니다.

**Web Worker — 메인 스레드를 살리는 한 줄.** `useWebWorker: true` 옵션 한 줄이 결정적이에요. 5~10초 걸리는 저가폰에서 메인 스레드를 막아두면 폼 UI가 그 시간 동안 굳습니다(스크롤, 탭, 입력 전부 정지). 워커로 빼면 메인 스레드는 멀쩡하고 사용자에게 "사진을 정리하는 중…" 인디케이터만 보여줄 수 있어요. 한 줄짜리 옵션이지만 모바일 UX가 통째로 달라집니다.

```ts title="components/apply/PhotoUpload.tsx"
out = await imageCompression(f, {
  maxSizeMB: PHOTO_COMPRESSION.maxSizeMB,               // 1MB 목표
  maxWidthOrHeight: PHOTO_COMPRESSION.maxWidthOrHeight, // 1920px
  useWebWorker: true,                                    // ← 결정적
  fileType: "image/jpeg",
  initialQuality: PHOTO_COMPRESSION.initialQuality,     // 0.85
});
```

**압축 실패 → 원본 폴백.** 라이브러리가 실패할 수 있는 케이스를 가정해야 해요(메모리 부족, 특이 포맷, 디코드 실패 등). 압축이 실패하면 _원본을 들고 다음 단계로_ 가고, 그 다음의 형식·용량 검증이 다시 판단합니다. 폼은 절대 막지 않아요. 사용자는 압축 실패와 무관하게 폼을 제출할 수 있어야 하니까요.

```ts title="components/apply/PhotoUpload.tsx"
try {
  out = await imageCompression(f, { /* ... */ });
} catch (err) {
  // 압축 실패 → 원본을 그대로. 다음 단계의 형식·용량 검증이 원본을 다시 본다.
  reportApplyError({ stage: "compress-failed", /* ... */ });
}
```

## 마무리 — 옮기지 않고 없앤 시간

오늘 한 일을 다시 짚으면 이렇습니다.

- [이전 글](/posts/direct-to-storage-upload)의 `Promise.all`과 `after()`는 시간을 **옮겼다**. 사용자의 체감 시간(=`/api/apply` 응답)은 줄었지만 4G의 98초는 그대로 남았다.
- 이번 글의 클라이언트 압축은 시간을 **없앴다**. 사용자 CPU 500ms로 81초를 회수했다. 시간 자체가 사라진 자리에선 그 위에서 무엇을 _옮길_ 필요도 없다.
- *주파수 도메인에서 블러와 압축이 같은 일을 한다*는 사실 — 블러 처리본이 80% 더 작아지는 부수효과로 다시 확인됐다. 의도하지 않은 보너스지만, 원리에서 보면 당연한 결과였다.

마지막으로 설계 메타포를 한 줄로 써보자면. **시간을 옮기는 도구와 없애는 도구는 같은 문제에 같이 들어가야 한다.** `Promise.all`/`after()`는 응답 시간을, 클라이언트 압축은 업로드 시간을 책임진다. 둘 다 있어야 *체감*과 _실제_ 양쪽이 풀린다. 그게 이번 두 글이 한 쌍인 이유입니다.
