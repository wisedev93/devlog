[이전 글](/posts/direct-to-storage-upload)의 마지막에 이렇게 적었습니다. `4G의 100초`는 일을 옮기는 것으로는 해결할 수 없는 문제이고, 문제를 해결하려면 업로드 바이트 자체를 줄여서 일을 없애야 한다고요.

이 글이 그 다음 이야기입니다. 같은 4G에서 98초 걸리던 업로드를 17초로 줄인 과정, 그 과정에서 들여다본 JPEG 압축의 내부, 그리고 우연히 발견한 블러-압축 시너지에 대한 기록입니다.

## Table of contents

## 왜 서버가 아니라 클라이언트인가

가장 먼저 떠오른 건 "서버가 받은 다음에 압축하면 되지 않나?"였습니다. 답은 아니었습니다. 서버 압축은 이번 문제를 두 가지 이유로 해결할 수 없었어요.

첫 번째, 4.5MB 한도가 그대로 남습니다. 압축은 서버가 받은 다음에 일어나므로, 사용자가 일단 원본(최대 40MB)을 서버에 보낼 수 있어야 해요. 한도를 넘는 순간 함수는 실행조차 못 합니다. [이전 글](/posts/direct-to-storage-upload)에서 서명 업로드 URL로 우회한 그 한도가 다시 살아나는 거죠.

두 번째, 업로드 시간이 그대로 남습니다. 4G에서 사진 4장을 보내는 데 98초가 걸렸습니다. 한도를 어떻게 우회해서 서버에서 압축한들, 사용자는 원본을 서버로 보내는 98초를 그대로 기다려야 해요. "서버가 압축한다"는 말은 결국 "원본 그대로 보낸다"는 뜻입니다.

그래서 압축은 사용자의 브라우저, 정확히는 사용자의 CPU에서 해야 했습니다. 사용자 한 명이 자기 폰에서 한 번 압축 비용을 내면 이후의 모든 단계(업로드, 저장, 블러)의 비용이 같이 줄어듭니다.

| 자리           | 4.5MB 한도 회피 | 업로드 시간 단축 | 비용       |
| -------------- | --------------- | ---------------- | ---------- |
| 클라이언트     | ✓               | ✓                | 사용자 CPU |
| 서버 (수신 후) | ✕ (한도 그대로) | ✕                | 서버 CPU   |
| CDN / 엣지     | ✕ (한도 그대로) | ✕                | 외부 비용  |

시간을 절약하려면 절약하려는 그 시점보다 앞에서 일을 끝내야 합니다. 그 "앞"이 클라이언트예요.

## 그러면 직접 업로드는 이제 필요 없나?

여기서 합리적인 의문이 하나 생겼습니다. 클라이언트에서 압축해 4장 합쳐 2.6MB쯤 되는 페이로드가 나온다면, 이전 글에서 만든 직접 업로드는 불필요해진 것 아닌가? FormData로도 4.5MB 안에 들어오는데?

측정값만 보면 맞는 말인데, 아키텍처로 보면 직접 업로드는 그대로 필요하다는 결론을 냈습니다. 이유는 세 가지입니다.

### 압축은 실패할 수 있고, 그때 원본이 그대로 전송된다

`PhotoUpload`의 `catch` 블록은 이렇게 되어 있습니다.

```ts title="components/apply/PhotoUpload.tsx"
try {
  out = await imageCompression(f, { /* ... */ });
} catch (err) {
  // 압축 실패 → 폼은 막지 않고 원본을 그대로 전송한다 (Android Chrome 의 HEIC 등)
  compressionFailed = true;
  reportApplyError({ stage: "compress-failed", /* ... */ });
}
```

압축이 실패하면 원본을 그대로 다음 단계로 넘깁니다. Android Chrome의 HEIC 디코드 실패, 저가폰의 메모리 부족, 특이한 EXIF로 인한 canvas 에러, 워커 초기화 실패 등 실패 경로는 다양해서 폴백은 필수예요.

문제는 그 폴백의 페이로드 크기입니다. 원본은 장당 10MB까지 허용되니 4장 모두 폴백이면 최대 40MB, 4.5MB의 9배입니다. 압축이 한 번 깨지는 순간 한도가 다시 살아나고, 이때 직접 업로드가 없으면 신청 자체가 통째로 실패합니다.

정리하면 압축은 성능 최적화고, 직접 업로드는 구조적 안전망입니다. 안전망이 없으면 성능 최적화가 한 번 실패할 때 폼이 통째로 무너져요. 안전의 근거는 압축이 잘 되는 best case가 아니라 압축이 실패한 worst case를 어떻게 처리하느냐입니다.

### 약속의 기준은 통계가 아니라 최악이어야 한다

같은 얘기를 다르게 표현하면 이렇습니다. 폼은 사용자에게 "장당 10MB까지 올릴 수 있다"고 약속합니다. 그 약속을 지킬 수 있는 아키텍처여야 해요. "보통은 압축이 잘 돌아서 0.65MB로 줄어든다" 같은 통계는 약속을 지킬 수 있는 근거가 못 됩니다. 압축이 깨지는 그 한 번에 약속도 같이 깨지니까요. 사용자 입장에서는 "신청이 안 됐다"가 전부고, 왜 안 됐는지는 관심 밖입니다.

상한선을 일반적인 상황으로 정당화하는 건 자주 부딪히는 함정입니다. "보통 이 정도 들어오니까 괜찮다"는 언젠가 깨지고, 깨지는 시점은 보통 제일 비용이 큰 순간이에요. 압축이 도입됐다고 직접 업로드를 들어내는 건 상한선을 도로 낮추는 후퇴입니다.

### 함수 실행 시간은 비용이다

마지막 이유는 비용 모델입니다. Vercel 서버리스 함수는 실행 시간 × 메모리로 청구되고, 함수는 요청 본문을 다 받을 때까지 살아 있습니다.

FormData로 사진을 함수에 통과시키면, 4G 사용자가 17초 동안 사진을 밀어 올리는 그 17초 내내 우리 함수도 대기 상태로 살아 있어요. CPU는 거의 안 쓰지만 메모리는 잡혀 있고 시간은 카운트됩니다. 직접 업로드는 그 17초를 Supabase Storage 인프라로 옮겨버립니다. 우리 함수는 1초 안에 끝나는 JSON 두 번(티켓 발급, row insert)만 처리하면 끝이에요. 트래픽이 작을 땐 안 보이는 차이지만, 동시 사용자가 늘수록 곱해지는 차이입니다.

덤으로 Supabase의 서명 업로드 URL은 resumable 업로드(TUS 프로토콜)를 지원합니다. FormData는 90%에서 연결이 끊기면 처음부터 다시 시작해야 하는데, 모바일에서는 이게 꽤 큰 차이예요.

### 서로 다른 레이어의 문제

| 도구            | 문제                                  | 레이어    |
| --------------- | ------------------------------------- | --------- |
| 직접 업로드     | 페이로드 한도 · 폴백 경로 · 함수 비용 | 구조      |
| 클라이언트 압축 | 전송 시간                             | 성능 / UX |

직접 업로드는 worst case를 받아내고, 압축은 best case를 만듭니다. 한쪽만 있어도 보통은 동작하지만, 문제가 생기는 순간 다른 쪽이 없으면 답이 없어요. 그래서 두 작업은 상호보완적인 작업입니다.

## 어떻게 줄어드나: 두 단계의 손실

브라우저에서 사진을 줄일 때는 성질이 다른 두 가지 손실이 함께 일어납니다.

### 1단계: 해상도 다운스케일

가장 단순한 손실입니다. 4032×3024(12MP) 사진을 1920px로 줄이면 픽셀 수가 약 1/6이 됩니다. 칸 자체를 줄이는 거고, 칸이 줄면 데이터도 줍니다.

브라우저는 이걸 `<canvas>`로 처리합니다. 라이브러리가 `Image`로 원본을 디코드하고, 더 작은 canvas에 `drawImage(..., dstW, dstH)`로 그립니다. 그리는 과정에서 브라우저의 bilinear/bicubic 보간이 일어나 적당히 안티앨리어싱된 다운스케일 결과가 나와요.

이 손실은 회복 불가능합니다. 줄어든 칸 안에 원본의 픽셀 단위 정보는 더 이상 없어요. 다만 폰 화면(보통 폭 700~1100px)으로 보기에 1920px은 이미 충분해서 사람 눈에는 차이가 거의 안 보입니다.

### 2단계: JPEG 양자화, 진짜 손실이 일어나는 곳

해상도를 줄여도 칸 안의 값을 그대로 저장하면 여전히 큽니다. JPEG는 여기서 진짜 압축을 합니다. 핵심은 세 단계예요.

먼저 DCT. JPEG는 이미지를 8×8 블록으로 자르고, 각 블록에 Discrete Cosine Transform을 적용해 픽셀 값들을 주파수 성분으로 바꿉니다. 결과는 똑같이 8×8개의 숫자지만 의미가 달라져요. 왼쪽 위는 낮은 주파수(블록 전체의 평균 톤), 오른쪽 아래로 갈수록 높은 주파수(섬세한 디테일)입니다.

![JPEG 8×8 블록의 DCT 결과 — 좌상단이 DC(블록 평균 톤), 우하단으로 갈수록 고주파 AC](/assets/blog/client-image-compression/jpeg-dct-block.svg)

여기까지는 손실이 없습니다. DCT는 가역 변환이에요.

다음이 양자화인데, 손실은 전부 여기서 일어납니다. 각 주파수 계수를 양자화 테이블의 값으로 나누고 반올림하는데, 테이블의 값이 고주파일수록 큽니다. 고주파일수록 거칠게 반올림한다는 뜻이에요.

![JPEG 양자화 테이블 (q85 근처) — 저주파는 작은 값으로 정밀하게, 고주파는 큰 값으로 거칠게 양자화한다](/assets/blog/client-image-compression/jpeg-quantization-table.svg)

그런데 왜 이게 사진에서는 티가 안 날까요? 사람 눈이 고주파 변화에 둔감하기 때문입니다. 같은 강도의 차이라도 저주파에 있으면 즉시 보이고(하늘이 갑자기 어두워짐), 고주파에 있으면 잘 안 보입니다(잔디 한 가닥의 테두리). JPEG는 인간 시각의 이 불균등한 민감도를 인코딩 전략으로 삼았습니다. 사람이 못 느끼는 만큼만 정보를 버리는 거죠.

마지막이 Huffman 코딩입니다. 양자화된 결과는 특히 고주파 영역에서 대부분 0이에요. 길게 늘어선 0을 run-length로 묶고, 자주 나오는 값에 짧은 코드를 주는 Huffman 인코딩을 적용합니다. 이 단계는 손실이 없는 효율적인 표기일 뿐입니다.

이때 `quality` 파라미터(0~1)를 통해서 양자화 테이블의 스케일링 계수를 정해줍니다. 품질이 낮을수록 양자화 값을 크게(더 거칠게) 만듭니다. q85가 안전 구간으로 자주 추천되는 건, 저주파는 여전히 정밀하게 보존되고 고주파만 충분히 거칠게 양자화돼서 사람이 못 느끼는 부분만 깎이기 때문입니다.

## 화질 저하, 진짜 안 보이나

이론은 그렇다 치고 실제로 안 보이는지는 두 가지로 봤습니다.

객관 지표로는 SSIM과 PSNR이 있습니다. PSNR(Peak Signal-to-Noise Ratio)은 픽셀 값의 평균 제곱 오차 기반으로, 40dB 이상이면 거의 구별 불가, 30dB 이하면 명확히 보입니다. SSIM(Structural Similarity Index)은 사람 시각에 더 가까운 지표로, 1.0이 완전 동일이고 0.95 이상이면 대부분의 사람이 차이를 못 봅니다. JPEG q85에서 일반적인 사진은 SSIM 0.97~0.99 범위에 들어옵니다. 본문에서 `initialQuality: 0.85`를 기본값으로 잡은 근거예요.

상황 적합성도 봤습니다. 우리 케이스는 폰으로 보는 사진입니다. 1920px 입력을 700~1100px 폰 화면에서 보는 상황인데, 이 재축소 자체가 추가적인 다운샘플링이라 압축 아티팩트가 한 번 더 평균화돼 사라집니다. 4K 모니터에서 100% 확대로 평탄한 면을 응시하지 않는 한 알아볼 일이 없어요.

## 측정

조건은 이전 글의 측정과 같게 맞췄습니다. `2544×2532`(약 3.92MB) JPEG 4장(기본 3 + 옵션 1), Chrome DevTools Network 스로틀로 회선 고정, 각 5회 중앙값. 기기는 Mac + Chrome입니다.

### 압축률

```
[compress] photo-1.jpg: 3.92MB → 0.65MB (83% 감소)
[compress] photo-1.jpg: ... (125ms 소요)
```

| 구간                      | 원본 평균 | 압축 후 평균 | 감소율 |
| ------------------------- | --------- | ------------ | ------ |
| 기본 사진 (`photos`)      | 3.92MB    | 0.65MB       | 83%    |
| 옵션 사진 (`optionPhoto`) | 3.92MB    | 0.65MB       | 83%    |
| 압축 자체 소요 (장당)     | —         | —            | 125ms  |

압축 자체 비용은 데스크탑 Chrome 기준 장당 125ms, 4장이면 500ms 정도입니다. 아래 단축 시간과 비교하면 이 500ms가 얼마나 싼 거래인지 보입니다.

### 업로드 시간

| 회선                | 압축 전 (median) | 압축 후 (median) | 단축    |
| ------------------- | ---------------- | ---------------- | ------- |
| no-throttle (Wi-Fi) | 1980ms           | 592ms            | **70%** |
| Fast 4G             | 98,346ms         | 17,250ms         | **82%** |

이게 핵심 숫자입니다. 4G 98초가 17초가 됐어요. 압축 비용 500ms를 내고 81초를 회수한 거래입니다. Wi-Fi에서도 70% 단축이라 회선에 무관하게 의미가 있고요.

이전 글의 병렬화 효과(Fast 4G 2%, Wi-Fi 21% 단축)와 비교하면 차이가 분명합니다. 일을 옮기는 것(병렬화)은 작은 보너스였고, 일을 없애는 것(압축)이 결정적이었습니다.

### Supabase Storage 실제 저장 용량

| 버킷                                     | 장당 평균 |
| ---------------------------------------- | --------- |
| `applicant-photos` (원본 압축본)         | 634KB     |
| `applicant-photos-blurred` (블러 처리본) | 122KB     |

브라우저에서 압축한 0.65MB(=665KB)짜리 JPEG이 Supabase에 634KB로 저장됐습니다. 클라이언트 측정값과 거의 일치하니 전송 중 변형 없이 그대로 도달했다는 뜻이에요.

그리고 블러 처리본을 보세요. 122KB. 원본 압축본 대비 80% 추가 감소입니다. 처음 봤을 때 측정을 의심했던 숫자고, 의심한 만큼 파볼 가치가 있었습니다.

## 블러가 파일을 또 줄인다

634KB가 블러 한 번에 122KB라니, 잘못 잰 줄 알고 다시 측정했는데 같은 숫자가 나왔습니다. 설명은 위의 JPEG 압축 원리로 돌아가면 됩니다.

가우시안 블러는 low-pass filter입니다. 정의상 고주파 성분을 제거해요. 우리 코드의 `PHOTO_BLUR_SIGMA = 25`는 1920px 이미지에 적용하기엔 매우 강한 블러라, 약 75픽셀(3σ) 이하의 디테일이 사실상 사라집니다. 결과물은 부드러운 색 그라데이션에 가까워요.

이걸 JPEG로 다시 인코딩하면 무슨 일이 일어나는지, 앞의 3단계를 다시 따라가 보면 명확합니다. DCT 단계에서는 8×8 블록의 거의 모든 픽셀이 비슷한 값이라 AC 계수들이 거의 0이 됩니다. 양자화 단계에서는 그 작은 AC 값들이 반올림으로 전부 0이 돼요. 64칸 중 63칸이 0이 되는 셈입니다. Huffman 단계에서는 0의 긴 런이 짧은 코드로 묶이면서 압축률이 폭발합니다.

다시 말해 블러는 JPEG가 버려야 할 후보(고주파)를 미리 다 0으로 만들어 둡니다. 양자화가 할 일이 없어지고 Huffman만으로 파일이 크게 줄어요.

80%라는 수치도 합리적인 범위입니다. 단색 1920px JPEG가 5~10KB인데, 심하게 블러됐지만 색과 밝기 그라데이션은 살아 있는 사진이 그 사이 어딘가인 122KB로 나온 거예요.

이건 의도한 결과가 아니라 부수효과였습니다. 블러는 서비스 특성상 필요해서 넣은 건데, 같은 작업이 Storage 용량까지 80% 깎아준 거죠. 주파수 도메인에서 보면 블러와 압축이 같은 일을 하기 때문에, 원리를 알고 나면 당연한 결과이기도 합니다.

## 지불한 비용들

압축이 무조건 이득은 아니라서, 지불한 비용도 확실히 해야합니다.

**저가폰의 CPU 시간.** 측정한 125ms는 데스크탑 Chrome 기준입니다. 실사용 환경은 더 느려서 최신 아이폰 1\~3초, 저가 안드로이드 5\~10초까지 봐야 해요. 4G에서 81초 아끼는 게 5~10초 압축 비용보다 크니 여전히 이득이지만, Wi-Fi에서 1.4초 아끼려고 10초 압축하는 건 손해입니다. 회선이 빠를수록 이득 폭이 줄어든다는 점은 의식해야 해요. 더 정밀하게 가려면 회선과 기기를 보고 압축을 건너뛰는 처리까지 할 수 있는데, 그 작업의 가치가 구현 비용 대비 작다고 봐서 이번엔 단순한 쪽을 택했습니다.

**HEIC on Android.** `browser-image-compression`은 HEIC를 마법처럼 변환해주지 않습니다. 내부적으로 `Image` → `canvas.drawImage` → `canvas.toBlob('image/jpeg')` 흐름을 타는데, 이건 브라우저가 HEIC를 디코드할 줄 알 때만 동작해요. iOS Safari는 네이티브로 지원해서 통과하고, Android Chrome은 지원하지 않아 실패 → 원본 폴백 → 형식 검증에서 거부로 흐릅니다. 안드로이드 사용자가 HEIC를 들고 오는 일 자체가 드물어 이걸로 충분하지만, 모든 HEIC를 처리하는 마법은 없다는 건 분명합니다.

**EXIF orientation.** 픽셀 단위로 재인코딩하면서 EXIF를 통째로 버리면 회전 정보도 같이 사라져 사진이 옆으로 누워 올라갑니다. 라이브러리가 orientation을 픽셀에 미리 반영한 뒤 EXIF를 버리는 식으로 자동 처리해줍니다. canvas로 직접 짠다면 EXIF 파싱과 회전 처리가 별도 작업으로 따라온다는 말이기도 해요.

**Web Worker.** `useWebWorker: true` 옵션이 결정적입니다. 압축에 5~10초 걸리는 저가폰에서 메인 스레드를 막아두면 그 시간 동안 폼 UI가 통째로 굳어요. 스크롤, 탭, 입력 전부요. 워커로 빼면 메인 스레드는 멀쩡하고, 사용자에게 "사진을 정리하는 중…" 인디케이터를 보여줄 수 있습니다. 한 줄짜리 옵션인데 모바일 UX가 갈립니다.

```ts title="components/apply/PhotoUpload.tsx"
out = await imageCompression(f, {
  maxSizeMB: PHOTO_COMPRESSION.maxSizeMB,               // 1MB 목표
  maxWidthOrHeight: PHOTO_COMPRESSION.maxWidthOrHeight, // 1920px
  useWebWorker: true,                                    // ← 결정적
  fileType: "image/jpeg",
  initialQuality: PHOTO_COMPRESSION.initialQuality,     // 0.85
});
```

**압축 실패 시 원본 폴백.** 메모리 부족, 특이 포맷, 디코드 실패 등 라이브러리가 실패할 케이스는 가정해야 합니다. 압축이 실패하면 원본을 들고 다음 단계로 가고, 그다음의 형식·용량 검증이 다시 판단합니다. 사용자는 압축 실패와 무관하게 폼을 제출할 수 있어야 하니까요.

```ts title="components/apply/PhotoUpload.tsx"
try {
  out = await imageCompression(f, { /* ... */ });
} catch (err) {
  // 압축 실패 → 원본을 그대로. 다음 단계의 형식·용량 검증이 원본을 다시 본다.
  reportApplyError({ stage: "compress-failed", /* ... */ });
}
```

## 마무리: 옮기지 않고 없앤 시간

이전 글의 `Promise.all`과 `after()`는 시간을 옮겼습니다. 사용자의 체감 시간은 줄었지만 4G의 98초는 그대로 남았어요. 이번 글의 클라이언트 압축은 시간을 없앴습니다. 사용자 CPU 500ms로 81초를 회수했고, 시간 자체가 사라진 곳에서는 무언가를 옮길 필요도 없어졌습니다.

블러 처리본이 80% 더 작아지는 부수효과는 주파수 도메인에서 블러와 압축이 같은 일을 한다는 사실의 확인이었습니다. 의도하지 않은 보너스지만 원리로 보면 당연한 결과였고요.

시간을 옮기는 도구와 없애는 도구는 **"같은 문제에 같이"** 들어가야 합니다. `Promise.all`/`after()`는 응답 시간을, 클라이언트 압축은 업로드 시간을 책임집니다. 둘 다 있어야 체감과 실제가 같이 해결됩니다. 그게 이전 글과 이 글이 한 쌍인 이유입니다.
