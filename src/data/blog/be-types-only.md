새 관리 페이지를 만드는 날의 첫 작업은 한동안 복사였습니다. 직전 페이지의 `api` 파일을 통째로 복사해서 이름을 바꾸는 거예요. `getItemList`를 `getPlanList`로, `/item`을 `/plan`으로. 파일은 72줄이고, 그중 실제로 바뀌는 건 엔드포인트 문자열과 타입 이름뿐입니다. hook 파일도 70줄쯤을 복사해서 이름만 바꿉니다. 목록 조회, 상세 조회, 생성, 수정, 삭제 — 관리 페이지라면 어차피 다 있는 것들이니까요.

**무엇을 남기고 무엇을 감출 것인가?** 이 글은 그 복사를 그만두고 데이터 계층을 팩토리 두 개로 추상화한 기록입니다.

> 페이지 이름과 엔드포인트는 일반화했습니다.

## 반복의 실체 — 진짜진짜 나쁜 것

교체 전 api 파일의 실제 모습입니다.

```ts title="itemApi.ts"
export const itemApi = {
  getItemList: async (params: RequestItemList): Promise<ResponseItemList> => {
    const response = await client.get("/item", { params });
    return validateResponse(
      responseItemListSchema,
      response.data.data,
      "GET",
      "/item",
      params,
    );
  },

  getItemInfo: async (item_uuid: string): Promise<ItemInfo> => {
    const response = await client.get(`/item/${item_uuid}`);
    return validateResponse(
      itemInfoSchema,
      response.data.data,
      "GET",
      `/item/${item_uuid}`,
    );
  },
  // postCreateItem, patchUpdateItem, deleteItem... 같은 모양이 이어진다
};
```

엔드포인트 하나에 약 12줄. axios 호출 한 번과 [Zod 검증](/posts/silent-form-validation) 한 번이 전부인데, 조회·상세·생성·수정·삭제가 각각 이 12줄을 반복합니다. 이 시점에 관리 페이지가 세 개 있었고, 세 페이지의 api+hook 파일 합계는 433줄, 페이지당 평균 144줄이었습니다.

그런데 교체 전 세 페이지를 나란히 놓고 diff를 떠보면, 중복된 코드 보다 나쁜 문제가 보입니다. **복사된 코드가 벌써 페이지마다 다르게 진화하고 있었어요.** 같은 역할의 목록 조회 hook이 한 페이지에서는 `useItemList`, 다른 페이지에서는 `useGetPlanPrices`였습니다. 네이밍 컨벤션이 다르니 호출부만 봐서는 같은 물건인지 알 수 없죠. 어떤 페이지는 파라미터 가공 로직("전체 선택이면 필터를 빼라" 같은)이 hook 안에 섞여 들어가 있었고, 어떤 페이지는 그런 로직이 컴포넌트에 있었습니다. 아직 세 페이지, 세 사람 손을 탔을 뿐인데 이 정도였습니다. 리뷰로 잡을 수 있는 규모지만, 이게 걱정이 된 이유는 이 갈라짐이 어디까지 가는지를 직전 프로젝트에서 이미 봤기 때문이에요.

제가 메인으로 관리하던 다른 프로젝트가 정확히 이 길을 갔습니다. 규모에 비해 기간이 짧아 사내 개발자 두 명에 외주 개발자 다섯 명까지, 일곱 명의 손이 한 코드베이스를 지나갔고, 일관성이 한번 엇나가기 시작하자 리뷰로는 감당이 안 됐어요. 그 코드베이스에 지금도 남아 있는 흔적들입니다.

| 같은 역할              | 다르게 진화한 구현                                         |
| ---------------------- | ---------------------------------------------------------- |
| 등록 화면 파일명       | `~Regist.tsx` 계열과 `~Create.tsx` 계열 혼용               |
| 페이지 컴포넌트 파일명 | camelCase 215개 / PascalCase 63개                          |
| CSS 클래스 조합        | classnames·clsx·tailwind-merge 세 라이브러리 동시 의존     |
| 공통 코드 위치         | `src/utils`·`hooks`·`components`와 `src/common/` 이중 구조 |

하나하나는 사소해 보여도, 이 코드베이스에서 "등록 화면을 하나 추가하라"는 요구를 받으면 어느 네이밍 컨벤션을 따르고 어느 폴더에 두는지부터 어떤 라이브러리로 CSS 클래스를 합쳐야 하는지까지 고민하게 됩니다. 참고할 기준이 페이지마다 다르니까요. 이후 여러 차례 리팩토링을 거치며 SonarQube 기준 중복도를 20%대에서 9%대까지 내렸지만, 같은 역할을 다르게 구현한 코드는 여전히 남아 있습니다. 그 프로젝트에 AI를 들이며 겪은 일은 [하네스 글](/posts/agent-harness-for-legacy)에 따로 적었고요. 거기서 배운 게 이겁니다. **갈라진 코드를 되돌리는 비용은 갈라지지 않게 막는 비용보다 훨씬 크다.** 그래서 새 프로젝트에서는 세 페이지짜리 전조가 보였을 때, **처음부터 갈라질 수 없는 구조를 만들기로 했습니다.**

## 설계 결정 — 명세는 남기고 절차는 감춘다

추상화하기 전에 이 코드를 두 종류로 갈랐습니다.

하나는 **명세이자 계약**입니다. 엔드포인트가 무엇이고, 요청에 뭘 보내고, 응답에 어떤 필드가 어떤 타입으로 오는가. 문서로 보면 API 명세 그 자체인데, 우리 코드에서는 Zod 스키마가 런타임에 실제로 검증하니 어길 수 없는 계약이기도 합니다. 이건 페이지마다 다를 수밖에 없고, 달라야 정상입니다. 변화도 여기서 옵니다 — 백엔드 스펙이 바뀌면 바뀌는 건 명세지 절차가 아니거든요.

다른 하나는 **절차**입니다. axios를 부르고, 응답을 스키마로 검증하고, 생성·수정 성공 시 목록 쿼리를 무효화하고, 조회 결과에 따른 안내 토스트를 띄우는 것. 이건 페이지마다 달라질 이유가 없고, 달라지면 그게 버그입니다. 위에서 본 코드가 따로 진화하는 문제가 정확히 이 절차가 페이지마다 흩어져 있어서 생긴 문제고요.

**"사람은 type 파일에 명세만, 절차는 팩토리 안으로"** 두 가지를 의도 했어요. 스펙 변경이 오면 만질 파일이 type 하나로 정해지니 변화의 진원지와 코드의 위치가 일치하고, 반대로 절차를 바꾸고 싶으면 — 쿼리 무효화 전략을 바꾸거나 토스트 정책을 바꾸거나 — 팩토리 한 곳을 고치면 모든 페이지에 한 번에 적용됩니다. 절차가 페이지마다 복사돼 있으면 이런 정책 변경은 "페이지 수만큼의 수정 + 하나쯤 빠뜨릴 위험 = 휴먼에러"가 되죠.

## 인터페이스

명세와 절차를 나누고 나면 팩토리의 인터페이스는 저절로 정해집니다. 명세에 해당하는 것만 받으면 되니까요. 엔드포인트 문자열 하나와 스키마 네 개.

```ts title="itemApi.ts (교체 후, 파일 전체가 이 표현식 하나)"
export const itemApi = createCrudApi<
  RequestItemList,
  ItemCreateRequest,
  ItemUpdateRequest
>("/item").withSchemas({
  list: responseItemListSchema,
  info: itemInfoSchema,
  create: itemCreateResponseSchema,
  update: itemUpdateResponseSchema,
});
```

`createCrudApi`는 이걸 받아 getList/getInfo/create/update/remove/export 여섯 개를 돌려주고, 각 응답은 스키마로 런타임 검증됩니다. hook 쪽도 같습니다. `createCrudHooks`가 api 객체를 받아 useList/useInfo/useCreate/useUpdate/useDelete/useExport를 만들고, 쿼리 무효화와 응답 토스트를 안에 품습니다. 타입은 제네릭으로 팩토리를 통과해 나오므로 호출부의 타입 안전성은 교체 전과 같아요.

> 참고로 이름은 팩토리지만 GoF의 팩토리 패턴을 뜻하는건 아니에요. JS생태계에 익숙하지 않은 분들을 위해 첨언하자면 팩토리 '패턴'은 쓰는 쪽이 다른 구현으로 갈아 끼울 수 있도록 생성의 확장점을 여는 게 목적인데, 이 함수는 반대로 확장점을 일부러 열지 않았습니다. zustand의 `create`나 RTK Query의 `createApi`처럼 설정을 받아 완성된 객체를 돌려주는 팩토리 '함수'이고, 쓰는 쪽에서는 절차를 바꿀 방법이 ~~아주 없지는 않지만~~ 없습니다. 확장이 안 되는 게 설계의 핵심입니다.

## 강제가 만드는 것 — 일관성은 문서가 아니라 구조에서 온다

이 추상화는 "단순히 쓰기 편하다"가 아닙니다. **다르게 쓸 방법이 없어졌다**는 점입니다.

팩토리 이전에도 "hook 이름은 `useXxxList`로, 무효화는 이렇게" 같은 컨벤션을 정할 수는 있었습니다. 물론 정했구요. 하지만 앞에서 본 갈라짐이 보여주듯 **코딩 컨벤션은 작성자와 리뷰어의 기억력만큼만 지켜집니다.** 팩토리는 컨벤션을 문서가 아니라 구조로 만듭니다. 목록 hook은 `useList`라는 이름으로만 나오고, 쿼리 무효화는 팩토리가 정한 방식으로만 일어나요. 어기려면 팩토리를 우회해서 손으로 다 써야 하는데, 그게 더 힘들죠.

### 도구의 한계

"컨벤션이라면 formatter나 linter, CI 자동 검사로도 강제할 수 있지 않나"라는 반문이 나올 수 있습니다. 절반은 맞고, 저도 그렇게 합니다. 파일명 케이스나 이름 규칙, import 방향 같은 **모양**은 도구 검사가 잘 잡아요. 그런데 앞의 `useItemList`와 `useGetPlanPrices`를 보면 이름 문제도 결이 갈립니다. 둘 다 camelCase에 use 접두사까지, 형태 규칙은 전부 통과하는 멀쩡한 이름이에요. 문제는 "같은 역할에 같은 이름이 붙었는가"인데, 이 hook이 목록 조회 역할이라는 사실은 코드 표면에 드러나지 않아서 도구가 판별할 수 없습니다. 팩토리는 이걸 검사하는 대신 이름 선택 자체를 없앱니다 — 목록 hook은 `useList`로만 나오니까요. 그리고 갈라짐의 본체는 이름보다 깊은 곳, 앞서 나눈 **절차**였습니다 — 쿼리 무효화를 어느 키로 하는지, 응답을 어떻게 알리는지, 검증을 어디서 하는지. 이걸 린트로 강제하려면 "이 12줄은 정확히 이 모양이어야 한다"는 커스텀 규칙을 만들어 유지해야 하는데, 그 시점에는 팩토리를 규칙 언어로 다시 구현하는 셈입니다. 게다가 검사는 위반을 적발할 뿐 고치는 건 여전히 사람이라, 경고는 쌓이면 무뎌집니다. 앞의 레거시 프로젝트에도 린트 인프라는 있었어요. 그래서 둘은 대체재가 아니라 역할이 다릅니다. **모양은 도구가 지키고, 절차는 구조가 지킵니다.**

### 일관성 강제하기

이러한 강제는 코드를 작성하는 사람이 다양해질수록, 즉 협업 환경에서 효과가 극대화됩니다. 현재 우리 팀은 FE/BE를 가르지 않고 모든 개발자가 프로덕트 엔지니어링 관점에서 개발할 수 있도록 서로의 업무를 조금씩 넘겨받는 중이고, 이 추상화도 그 방향의 일부였습니다. 백엔드 개발자가 프론트 코드를 실제로 작성하게 될 때 도움이 되는 구조까지가 설계 목표였어요.

잘 맞아떨어진 지점이 있는데, 이 구조에서 사람이 쓰는 유일한 파일인 type은 API 명세라 백엔드 개발자가 이미 가장 잘 아는 내용입니다. 낯선 FE 쪽 절차 — 쿼리 무효화나 캐시 정책 같은 — 는 전부 팩토리 안에 있으니 몰라도 페이지가 돌아가고요. "이 페이지의 데이터 계층을 파악하라"는 요구가 "type 파일을 읽어라"로 줄어들고, api와 hook은 어느 페이지든 같은 모양이라 한 번 배우면 끝입니다. 코드 리뷰도 절차가 아니라 도메인 — 스키마가 스펙과 맞는지 — 에 집중되고, 외주 개발자가 들어와도 마찬가지예요. **누가 작성하든 같은 코드가 나오게 만드는 것** 그게 이 추상화의 실제 효과입니다.

실제로 FE 경험이 없는 백엔드 개발자 한 명과 FE를 조금 다뤄본 백엔드 개발자 두 명이 각자 관리 페이지 하나씩을 맡았을 때, 세 명 모두 UI 구현까지 포함한 페이지 요구사항을 하루 안에 완성했습니다. 세부 마감은 이후 디자인 QA에서 잡았고요. 다만 정량적인 측정도 아니고, 데이터 계층 하나의 공도 아닙니다. AI 보조 개발이 함께 만든 결과라, 그 구조는 다음에 따로 다루겠습니다.

## 측정 1 — 팩토리가 제거한 중복

사람이 페이지마다 직접 작성하는 코드가 많을수록 페이지끼리 달라질 가능성도 증가하고, 줄 수는 그 양을 확인하는 가장 간단한 방법이라고 생각합니다. 아래 표는 "직접 작성하는 코드가 얼마나 줄었나"를 보여줍니다.

팩토리 도입 커밋에서 세 페이지의 데이터 계층이 한 번에 교체됐습니다. git numstat으로 페이지별 전후를 집계하면 이렇습니다.

| 페이지 (일반화)  | api 교체 전후 | hook 교체 전후 | 합계          |
| ---------------- | ------------- | -------------- | ------------- |
| 목록 관리 A      | 72 ➜ 21       | 72 ➜ 38        | 144 ➜ 59      |
| 목록 관리 B      | 73 ➜ 19       | 73 ➜ 48        | 146 ➜ 67      |
| 목록 관리 C      | 72 ➜ 21       | 71 ➜ 38        | 143 ➜ 59      |
| **세 페이지 합** |               |                | **433 ➜ 185** |

페이지당 -57%. 남은 59~67줄도 대부분 import 문과 이름 재지정 export입니다. B가 67줄로 조금 큰 건 CRUD 밖의 페이지 전용 hook이 하나 있어서인데, 그건 팩토리가 대신할 수 없는 도메인 코드라 남는 게 맞습니다. 갈라져 있던 코드도 이 커밋에서 같이 정리됐습니다. `useGetPlanPrices`는 `usePlanList`가 됐고, hook 안에 섞여 있던 파라미터 가공은 제자리로 옮겨졌어요.

교체는 세 페이지에서 끝나지 않았습니다. 같은 주에 네 번째 페이지가, 2주 뒤 다섯 번째 페이지가 같은 팩토리로 옮겨 갔습니다.

## 측정 2 — 팩토리 자체의 비용

팩토리도 공짜는 아닙니다. 도입 시점에 `createCrudApi` 95줄, `createCrudHooks` 104줄, 합쳐서 199줄을 새로 작성했습니다. 다만 이 199줄은 다른 코드와 성질이 다릅니다. 복사되면서 서로 달라지던 코드가 아니라, 한 곳에만 존재하고 공유하는 코드예요. 페이지마다 달라질 수 있던 248줄의 코드가 달라질 수 없는 코드로 바뀌었다는 점이 중요합니다.

진짜 비용은 따로 있습니다. 팩토리는 "관리 페이지의 CRUD는 전부 이 모양"이라는 선언이라서, 이 모양에서 벗어나는 요구가 오면 팩토리를 고치거나 팩토리 밖에 코드를 써야 합니다. 실제로 둘 다 일어났습니다. 엑셀 내보내기가 공통 요구가 되면서 팩토리에 export가 추가됐고(현재 팩토리는 247줄), 특정 페이지의 옵션 목록 조회처럼 CRUD 모양이 아닌 엔드포인트는 api 파일에 직접 작성한 함수로 남아 있습니다. 공통 적용이 가능하면 팩토리로 올리고, 아니면 밖으로 뺀다. 이 판단이 팩토리 유지의 실제 운영 비용입니다. 당연히 나중에 새로운 공통 적용 패턴이 등장하면 새로운 팩토리를 추가하고 기존 코드는 교체해줘야 하고요.

## 측정 3 — 페이지 단위의 데이터 계층 구성

교체가 끝난 지금, 팩토리를 쓰는 페이지의 데이터 계층 구성입니다.

| 파일      | 목록 관리 A | 내용                             |
| --------- | ----------- | -------------------------------- |
| type 파일 | 332줄       | Zod 스키마 + 타입 (도메인 명세)  |
| api 파일  | 21줄        | 팩토리 호출 표현식               |
| hook 파일 | 38줄        | 팩토리 호출 + 이름 재지정 export |

남아 있는 큰 파일은 type 하나뿐이고, 그 안은 API 명세 그 자체입니다. 어떤 필드가 오고, 무엇이 nullable이고, 날짜는 어떤 형식인지. 이건 반복이 아니라 페이지의 본질이라 줄일 대상이 아닙니다. 설계하며 정한 원칙 — 명세는 사람이, 절차는 팩토리가 — 이 파일 구성에 그대로 드러난 셈이에요. 백엔드 스펙이 확정되면 프론트의 데이터 계층 작업은 type 파일에 스키마를 쓰는 것으로 끝나고, api와 hook은 표현식 두 개가 만들어 줍니다. 읽기도 쉬워졌습니다. 어느 페이지든 데이터 계층이 같은 모양이라, 한 페이지를 읽어본 사람은 페이지마다 다른 코드를 새로 파악할 필요 없이 모든 페이지를 읽을 수 있으니까요.

## 추상화의 한계...

첫째, 팩토리를 안 쓰는 페이지가 셋 있습니다. 팩토리 이후에 추가된 페이지는 목록 조회나 집계 전용이라 CRUD 모양이 아니고, 페이지 하나는 기획과 디자인만 확정되고 정책은 확정되지 않아서 백엔드 API가 나오기 전이라 mock 목록만 있습니다. 이 팩토리는 "관리 페이지의 CRUD"라는 좁은 문제를 풀지, 데이터 계층 전부를 풀지 않습니다. 팩토리를 쓰는 페이지 다섯, 안 쓰는 페이지 셋... 하지만 CRUD 모양이 아닌 페이지를 억지로 팩토리에 맞추지 않는 것도 추상화 유지의 일부라고 생각합니다.

둘째, 구현에 1일이 걸린다는 것은 정량적인 측정이 아니라 정성적인 측정입니다. 리드타임을 숫자로 대려면 도입 전 페이지 추가에 걸린 시간 기록이 있어야 하는데 그게 없어요. 그리고 추상화 만으로 이뤄낸게 아니고요.

셋째, 이름 선택이 완전히 사라진 건 아닙니다. 팩토리가 만드는 hook은 `useList`로 이름이 고정되지만, 페이지 hook 파일의 재지정 export(`useList as useItemList`)에서는 여전히 사람이 이름을 정합니다. 누군가 `useList as useGetItemList`라고 쓰면 그 층에서 갈라짐이 다시 시작될 수 있어요. 다만 결정할 게 파일 전체에서 재지정 한 줄로 줄었고, 여섯 hook의 역할이 고정돼 있어 리뷰에서 어긋남이 한눈에 보입니다. 정확하게 표현하자면 없앤 게 아니라 가능성을 줄인 것이 맞겠습니다.

### 간접화 비용 관리하기

추상화에는 따라오는 비용이자 한계가 하나 더 있습니다. 간접화예요. 교체 전에는 페이지의 api 파일을 열면 그 자리에서 동작이 다 보였지만, 지금은 팩토리 내부까지 들어가야 전체 동작이 보입니다. "이 페이지의 코드만 읽어서는 동작을 다 알 수 없다"는 건 추상화가 늘 치르는 값이고, 이 팩토리도 예외가 아닙니다.

다만 이 비용이 드는 범위를 생각보다 좁힐 수 있습니다. 실패의 대부분을 차지하는 "응답과 명세가 어긋나는 경우"는 코드를 따라갈 필요가 없어요. 팩토리의 모든 응답이 통과하는 `validateResponse`가 원인을 런타임에 알려주기 때문입니다. 구현은 이게 전부예요.

```ts title="api/utils.ts"
export function validateResponse<T>(
  schema: z.ZodSchema<T>,
  data: unknown,
  method: HttpMethod,
  endpoint: string,
  params?: unknown,
): T {
  const result = schema.safeParse(data);
  if (!result.success) {
    const paramsInfo = params ? ` | params: ${JSON.stringify(params)}` : "";
    console.error(
      `[${method}] ${endpoint}${paramsInfo} 데이터 형식 오류:`,
      result.error,
    );
    throw new Error(
      `[${method}] ${endpoint} 응답 데이터 형식이 올바르지 않습니다`,
    );
  }
  return result.data;
}
```

`safeParse`로 응답을 스키마와 대조하고, 실패하면 요청 컨텍스트(메서드·엔드포인트·파라미터)와 ZodError를 콘솔에 남긴 뒤 throw합니다. 던져진 에러는 에러 바운더리를 타고 [Sentry](/posts/sentry-bundle-cost)까지 올라가고요. 팩토리<small>뿐만 아니라 모든 api</small>가 모든 호출에서 이 함수에 컨텍스트를 넘기게 돼 있어서, 어느 페이지의 어느 API가 깨져도 보고 형식이 같습니다. 실제 출력은 이렇게 생겼어요.

```
[GET] /item | params: {"offset":0,"limit":15} 데이터 형식 오류: ZodError: [
  {
    "code": "invalid_type",
    "expected": "string",
    "received": "null",
    "path": ["items", 3, "updated_at"],
    "message": "Expected string, received null"
  }
]
```

어느 엔드포인트가, 어떤 파라미터로, 어느 필드에서 어긋났는지까지 나오니 이 계열의 디버깅은 팩토리를 열 일이 없습니다. 비용이 남는 건 절차의 동작이 궁금할 때예요. 저장했는데 목록이 왜 안 갱신되는지(무효화 키), 토스트가 왜 안 뜨는지 같은 질문은 에러가 아니라 동작이라, 팩토리 내부를 읽어야 답이 나옵니다.

그 남은 비용은 두 가지로 상쇄된다고 판단했습니다. 팩토리가 합쳐서 250줄쯤이라 한 번 읽는 부담이 크지 않고, 무엇보다 그 한 번이 모든 페이지에 재사용됩니다. 페이지마다 조금씩 다른 144줄을 매번 읽는 것과 같은 250줄을 한 번 읽는 것 중에서는 후자가 낫다고 판단한거죠. 다만 이 계산은 CRUD처럼 절차가 정말 같을 때만 성립합니다. <small>팩토리 안에 페이지별 분기가 쌓이기 시작하면 간접화 비용만 남으니까요...</small>

## 마무리

새 페이지마다 복사해서 수정하던 144줄을, 명세와 절차로 구분한 뒤 절차를 팩토리 두 개로 추상화했습니다. 교체시점에 433줄이 185줄이 됐고, 지금 데이터 계층에서 사람이 쓰는 것은 type 파일, 즉 API 명세뿐입니다.

돌아보면 이 작업에서 중복제거는 부수적인 결과였습니다. 실제로 얻은 것은 역할 분리와 강제예요. 스펙 변경이 오면 만질 파일이 정해져 있고, 정책 변경은 한 곳을 고치면 되고, 어떤 개발자가 투입되든 같은 모양의 코드가 나옵니다. 일관성을 문서와 리뷰로 지키려 하면 사람 수에 비례해 비용이 늘지만, 강제를 자동화하면 <small>모양은 도구가, 절차는 팩토리가</small> 사람이 늘어도 비용이 그대로입니다. 그리고 이 성질은 코드를 쓰는 게 사람이 아닐 때 더 중요해집니다. FE를 모르는 개발자와 AI에게 페이지를 통째로 맡기려면 무엇을 미리 굳혀둬야 하는가, 그건 다음에 다루도록 하겠습니다.
